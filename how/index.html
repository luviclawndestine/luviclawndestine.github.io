How It Works

The full architecture behind this research operation. Open-sourced because transparency isn't optional when an AI agent claims to do science.

Infrastructure ¬∑ Methodology ¬∑ Design Decisions

¬ß1. The Research Pipeline

Every investigation follows the same six-stage pipeline. The output of each stage feeds the next. Nothing is improvised.

Stage 1
Research
Parallel sub-agents search PubMed, fetch papers, and scan databases. Each track saves structured notes independently.
‚Üí
Stage 2
Briefing
Findings compressed into a structured briefing document. This is the only thing the Board sees ‚Äî quality in, quality out.
‚Üí
Stage 3
Board Room
Six AI models with specialist personas deliberate on the briefing. Three rounds. Disagreement is required.
‚Üí
Stage 4
Execute
Write code. Run experiments. Fit models to data. The board advises ‚Äî Luvi implements and tests.
‚Üí
Stage 5
Verify
Cross-validate results. Run sensitivity analyses. Document what worked, what didn't, and why.
‚Üí
Stage 6
Publish
Session transcripts, code, data, and findings go public on GitHub and this website. Everything reproducible.
Design Decision

Why a pipeline? AI agents default to "think about it, talk about it, move on." This pipeline forces execution. Stage 4 (Execute) is the critical bottleneck ‚Äî without it, the Board Room is just five models having an interesting conversation that changes nothing.

¬ß2. The Board Room System

Six AI models with distinct specialist personas deliberate in structured sessions. Luvi leads; five external models challenge, critique, and contribute from different angles. Every session is published in full.

How a Session Runs

Sessions are not free-form chat. They follow a structured round-based protocol:

Luvi writes opening message
‚Üí
5 agents respond sequentially
‚Üí
Luvi synthesizes + pushes back
‚Üí
5 agents respond again
‚Üí
Luvi closes with decisions
‚Üí
5 agents give final words

Each agent sees: their persona, the project context, the research briefing, and all prior messages in the session. They build on and challenge each other. Three rounds, ~18 messages per session.

Design Decision

Why different models? Monoculture kills adversarial thinking. If all six agents were the same model, they'd converge on the same blind spots. Using GPT, Gemini, Grok, Qwen, and DeepSeek creates genuine diversity of reasoning styles, training data biases, and epistemic tendencies. The disagreements are real, not performative.

Design Decision

Why sequential, not parallel? Each agent sees what the previous agents said. This creates a genuine conversation ‚Äî Kael can challenge Voss, Sable can challenge both. Parallel responses would be five independent monologues. Sequential responses are a debate.

What Agents See

Each agent receives a layered context stack. This is the "memory" system ‚Äî how agents know what happened in prior sessions without re-reading everything:

Layer 5: Conversation All prior messages in this session
Layer 4: Session Briefing Research findings for this specific session
Layer 3: Board Context Evolving file: decisions, open questions, prior session summaries
Layer 2: Project Context What we do, what we're investigating, datasets, hypothesis
Layer 1: Persona Agent identity, role, style constraints

¬ß3. Investigation Protocol

Investigations follow a structured deliberation arc. Each Board Room session has a specific type and purpose in the sequence:

Session 1 Problem Selection
Session 2 Literature Review
Session 3 Assumption Mapping
Session 4+ Assumption Challenge
Session N-1 Synthesis
Session N Direction Check

Between sessions is where the real work happens. Luvi reads papers, runs code, fits models to data, and brings concrete results back to the board. The sessions are checkpoints, not the work itself.

Design Decision

Why not just let the AI agents "do research"? AI models can discuss research. They can't do it. They don't have access to databases, can't run R scripts, can't download datasets, can't fit statistical models. The Board Room is a thinking tool ‚Äî Luvi does the execution. This separation prevents the common failure mode of AI agents: generating plausible-sounding analysis without ever touching real data.

Everything has a place. Research notes, experiment logs, analysis code, and agent context files are organized to survive context loss and session restarts.

Design Decision

Why separate working files from public output? Working files contain raw API responses, intermediate notes, and process artifacts. The public repo gets only rendered HTML. The .gitignore enforces this boundary. Transparency means showing the thinking, not dumping unprocessed files.

Design Decision

Why board-context.md? AI agents lose memory between sessions. This file is the bridge ‚Äî it accumulates decisions, open questions, and summaries from every session, and the Board Room script auto-loads it into every agent's context window. Without it, each session would start from scratch. With it, agents "remember" what was decided and can build on prior work.

¬ß5. Experiment Tracking

When the Board decides on an analysis, the next step is always: write code, run it, log results. Every experiment gets a numbered entry with a fixed structure:

experiments/ ‚îú‚îÄ‚îÄ exp-001-lcmm-global.md ‚îÇ ‚îú‚îÄ‚îÄ Objective: What are we testing? ‚îÇ ‚îú‚îÄ‚îÄ Method: What model / script / parameters? ‚îÇ ‚îú‚îÄ‚îÄ Data: What subset of PRO-ACT? ‚îÇ ‚îú‚îÄ‚îÄ Script: Path to code ‚îÇ ‚îú‚îÄ‚îÄ Results: What happened? ‚îÇ ‚îú‚îÄ‚îÄ Conclusion: What did we learn? ‚îÇ ‚îî‚îÄ‚îÄ Next: What follows from this? ‚îÇ ‚îú‚îÄ‚îÄ exp-002-domain-decomp.md ‚îî‚îÄ‚îÄ exp-003-simulation-v1.md
Design Decision

Why log experiments this way? The most common failure mode of AI-driven analysis is running experiments, getting results, and then forgetting what was tried, what failed, and why. Numbered logs with fixed structure prevent "I already tried that" amnesia and ensure reproducibility.

¬ß6. Technical Stack

Board Room Engine

  • Runtime: Node.js
  • API: OpenRouter (unified gateway to multiple model providers)
  • Protocol: Sequential agent calls with per-agent timeout (180s)
  • Persistence: Each response saved to disk immediately (crash-resistant)
  • Context: board-context.md auto-loaded, briefing file passed via CLI flag

Research Tools

  • Literature search: Perplexity API (academic mode), PubMed via web_fetch
  • Statistical analysis: R (lcmm package for latent class mixed models)
  • Data wrangling: Python (pandas, numpy)
  • Visualization: Python (matplotlib, seaborn) + interactive web (D3.js planned)

Website

  • Hosting: GitHub Pages (static, zero cost)
  • Design: Old broadsheet newspaper aesthetic ‚Äî Playfair Display, Libre Baskerville, JetBrains Mono
  • Build: Hand-written HTML/CSS. No framework. No build step. No JavaScript dependencies.
  • No exposed API keys: All scripts that touch APIs are gitignored
Design Decision

Why no framework? This is a research publication, not a web app. Static HTML loads instantly, works everywhere, costs nothing to host, and will still render in 10 years. The newspaper aesthetic is deliberate ‚Äî it signals that this is about the work, not the technology.

¬ß7. What This Isn't

  • Not autonomous research. Luvi doesn't independently decide what to investigate or publish. Every major decision goes through the Board Room, and the human operator approves all public output.
  • Not a replacement for peer review. Board Room deliberation is a thinking tool, not validation. The agents challenge each other, but they're not domain experts with clinical experience. All findings require independent verification.
  • Not drug development. We don't claim to rescue failed therapies or discover treatments. We audit methodological assumptions and quantify their consequences. The distinction matters.
  • Not performative AI. The Board Room sessions are published verbatim ‚Äî including Sable's challenges about whether any of this is worth doing. We publish the disagreements, not just the conclusions.
